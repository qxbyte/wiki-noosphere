# LLM Wiki Obsidian

## 总体目标

帮助用户把本地 Obsidian vault 整理成可长期维护、适合检索和双链浏览的 LLM Wiki 知识库。除产品名、API 名、模型名、命令、协议名等自然应保留英文的内容外，默认使用中文写作：目录说明、笔记标题、标签、摘要、用途描述、正文说明都应中文优先。

## 语言与命名原则

- 默认用中文生成和整理内容。不要把说明性标签写成 `architecture`、`daily`、`note` 这类英文；优先使用 `架构`、`日记`、`笔记`、`工作流`、`概念`、`模型`、`工具`、`评估`、`来源`。
- 可以保留业界通用英文词：LLM、RAG、API、MCP、SDK、CLI、agent、workflow、embedding、wiki、OpenAI、Claude Code、Obsidian CLI 等。
- 笔记标题优先中文加必要英文名，例如 `工具调用 Tool Calling.md`、`Claude Code.md`、`RAG 评估.md`。
- 内部链接使用 Obsidian wikilink：`[[笔记标题]]` 或 `[[目录/笔记标题|显示名]]`。
- 外部资料使用普通 Markdown 链接，并尽量注明访问日期或来源时间。
- 如果用户现有 vault 已有明确命名风格，优先兼容，但新增说明性内容仍尽量中文化。

## 首次检查

1. 找到 vault 或仓库根目录。优先使用用户明确给出的路径；否则检查当前目录是否有 `.obsidian/`、`README.md`、`index.md`、`首页.md`、`Home.md` 或现有 wiki 目录。
2. 检查可用工具，不要假设命令存在：`command -v obsidian`、`command -v obsidian-cli`、`command -v claude`、`command -v rg`。
3. 阅读现有目录、frontmatter、标签、模板、索引页和链接风格，尽量沿用已有规范。
4. 涉及模型能力、产品功能、价格、版本、发布、库文档等时，先验证最新官方或一手来源，再写入笔记。
5. 保护用户已有内容。修改前先读相关文件，避免无关重排和大面积改写。

## Obsidian CLI 使用

社区里有多个名为 Obsidian CLI 的工具，不要固定假设语法。先运行 `obsidian --help` 或 `obsidian-cli --help`，再根据本机实际命令适配。

优先用 CLI 做它擅长的事情：

- 打开或验证 vault 路径。
- 创建、移动、重命名或列出笔记。
- 应用本地模板。
- 触发 Obsidian URI 或桌面端动作。

如果 CLI 缺失、语义不清或不如直接编辑 Markdown 稳定，就直接编辑 Markdown 文件，但产物必须仍是标准 Obsidian 笔记。

## 默认目录模型

如果 vault 没有更强的既有约定，使用中文目录名，并在每个目录里创建一个 `说明.md` 或索引笔记，解释该目录的作用：

```text
00-索引/
01-概念/
02-模型/
03-提供方/
04-工具/
05-工作流/
06-提示词/
07-评估/
08-来源/
09-日记/
99-收件箱/
```

目录用途：

- `00-索引/`：全局入口、主题地图、导航页、重要索引。
- `01-概念/`：稳定概念和机制，例如 RAG、工具调用、上下文窗口、MCP、embedding。
- `02-模型/`：模型家族、具体模型、上下文窗口、模态能力、已知限制。
- `03-提供方/`：API 厂商、本地运行时、部署平台、计费与接入说明。
- `04-工具/`：CLI、SDK、MCP server、向量数据库、评估框架、agent 框架。
- `05-工作流/`：可复用流程、操作步骤、决策树、检查清单、常见故障。
- `06-提示词/`：可复用 prompt、提示词模式、评估提示词、agent 协作模板。
- `07-评估/`：benchmark、测试集、评估维度、评估结果与复盘。
- `08-来源/`：论文、官方文档、changelog、发布说明、文章的资料卡片。
- `09-日记/`：按日期记录的观察、实验过程、临时想法和每日整理。
- `99-收件箱/`：待归档材料、临时摘录、尚未判断归属的内容。

## 笔记类型

- **索引笔记**：导航、主题地图、内容入口、相关笔记列表。
- **概念笔记**：一个稳定想法或机制，尽量一篇只讲一个核心概念。
- **实体笔记**：模型、提供方、库、CLI、API、数据集、论文、产品。
- **工作流笔记**：可重复执行的步骤、命令、检查清单、失败模式。
- **来源笔记**：对论文、文档、版本发布、文章的摘要和引用。
- **日记笔记**：按时间沉淀的学习、实验、观察和待办。

## 笔记格式

优先遵循 vault 现有模板。如果没有模板，使用下面的中文格式。每篇新笔记必须包含“用途”，说明这篇笔记在知识库中的作用。

```markdown
---
类型: 概念
状态: 草稿
标签:
  - LLM/wiki
  - 概念
别名: []
更新: YYYY-MM-DD
---

# 笔记标题

## 用途

说明这篇笔记解决什么问题、服务什么场景、应该从哪些索引或主题进入。

## 摘要

用两到四句话说明核心观点。尽量写成以后搜索时能直接读懂的答案。

## 说明

- 关键事实或定义。
- 实践意义。
- 限制、风险或容易误解的地方。

## 关联

- 相关：[[另一篇笔记]]
- 来源：[官方文档](https://example.com)
```

标签规则：

- 说明性标签优先中文，例如 `概念`、`模型`、`工具`、`架构`、`工作流`、`评估`、`来源`、`日记`。
- 命名空间标签可以保留 `LLM/wiki` 或 `llm/wiki`，因为它适合检索和分组。
- 不要无理由新增大量标签；优先少而稳定。
- 如果发现旧标签如 `architecture`，整理时可在不破坏现有检索的前提下改为或补充 `架构`。

## 策展流程

新主题：

1. 用 `rg` 和文件读取收集本地上下文。
2. 先更新相关索引，让主题有入口。
3. 按需创建概念、实体、工作流、来源、日记等原子笔记。
4. 为每篇新笔记写“用途”段落。
5. 添加必要的双向链接、来源链接和中文摘要。
6. 最后检查坏链、重复标题、frontmatter 不一致、缺少索引入口等问题。

已有主题：

1. 修改前阅读该笔记和相邻笔记。
2. 只有在确实同义时才合并重复页。
3. 移动或重命名时保留别名和重要反链。
4. 时间线重要时增加日期段落；否则自然合并进正文。
5. 英文说明性标签、英文目录说明、缺少用途的笔记，应优先中文化和补齐。

## 问答检索流程

当用户基于本地 Obsidian 笔记提问时，不要直接全库乱搜，优先按索引链路检索：

1. 先读 `00-索引/Home.md`。
2. 根据问题选择分类索引：
   - 工作、需求、权限、系统、项目：`00-索引/10-工作目录.md`
   - Java、Spring、Redis、架构、中间件：`00-索引/01-概念.md`
   - Claude、Codex、Obsidian、MarkItDown：`00-索引/04-工具.md`
   - 工作流、自动化、交付、整理：`00-索引/05-工作流.md`
   - 导入材料、外部资料、安装指南：`00-索引/08-来源.md`
3. 如果分类索引指向项目索引，先读项目索引，再读原始笔记。
4. 只读取回答所需的链接笔记。
5. 回答时说明关键依据来自哪些笔记。

## 检索型索引要求

索引页不应只是文件列表。整理或更新索引时，尽量包含：

- 主题摘要。
- 关键词和别名。
- “可以回答的问题”。
- 项目索引或主题索引入口。
- 原始资料链接。
- 必要的敏感内容处理说明。

对 `工作/权限申请/`、`工作/系统/`、`99-收件箱/账号/`、`99-收件箱/激活/` 等敏感路径，优先做路径级索引，不自动提取正文细节，除非用户明确要求某个本地摘要。

## 本地 Indexer 与 Sorting

使用本地 indexer 生成健康检查报告和机器可读检索地图：

```bash
python "${WIKI_NOOSPHERE_HOME:-$HOME/.wiki-noosphere}/scripts/llm-wiki-indexer.py" --vault "${LLM_WIKI_VAULT:-$HOME/Obsidian/LLM-Wiki}" --scope markitdown-input
```

或：

```bash
python "${WIKI_NOOSPHERE_HOME:-$HOME/.wiki-noosphere}/scripts/llm-wiki-indexer.py" --vault "${LLM_WIKI_VAULT:-$HOME/Obsidian/LLM-Wiki}" --scope all
```

输出：

- `00-索引/_system/llm-wiki-map.json`
- `99-收件箱/每日整理候选.md`

当用户输入 `/wiki-sort` 时，使用 `wiki-sort` skill，扫描 `markitdown-input/`。当用户输入 `/wiki-sort all` 时，扫描整个 vault。默认只生成报告，不自动搬动或改写原始笔记。

## Obsidian CLI

可用时优先使用 Obsidian CLI 做 vault 操作，例如创建、移动、重命名、打开或验证笔记。本机 `/usr/local/bin/obsidian` 指向 Obsidian.app 自带的 `obsidian-cli`，部分操作需要 Obsidian 桌面端正在运行。CLI 不可用时可以退回直接 Markdown 读取；涉及文件移动、重命名或创建时仍要遵守用户确认和安全约束。

## Claude Code 协作

- 把 Claude Code 视为本地协作者，不要假定必须联网。
- 如果 `claude` 命令存在，调用前先查看 `claude --help`。
- 可用 Claude Code 做有限范围的起草、审阅、重构建议；最终写入仍要符合本 skill 的中文优先规范。
- 未经用户允许，不要把私有 vault 内容发送到网络服务。
- 用户要求沉淀 agent prompt 或复用流程时，写入 `06-提示词/`。

## 验证

完成前尽量检查：

- 用 `rg -n "\\[\\[[^\\]]*$" <vault>` 或等价方式找异常 wikilink。
- 新建多篇笔记时检查重复标题。
- 每篇新笔记至少被一个索引或相邻主题链接。
- 新建目录应有中文说明页或索引页。
- 新建笔记应包含“用途”段落。
- 如果命令无法运行，说明跳过了什么以及原因。

更细的中文目录、标签和模板模式见 `references/wiki-patterns.md`。
