# LLM Wiki 中文整理模式

## 目录职责

- `00-索引/`：知识库入口、主题地图、常用导航、重要笔记列表。
- `01-概念/`：稳定概念、机制、术语和原理。
- `02-模型/`：模型家族、具体模型、能力边界、上下文窗口、模态支持。
- `03-提供方/`：API 厂商、本地运行时、部署平台、计费、接入方式。
- `04-工具/`：CLI、SDK、MCP server、向量数据库、评估库、agent 框架。
- `05-工作流/`：可复用流程、检查清单、命令序列、故障排查。
- `06-提示词/`：prompt 模板、提示词模式、评估提示词、agent 协作模板。
- `07-评估/`：benchmark、评测维度、测试集、结果记录、复盘。
- `08-来源/`：论文、官方文档、changelog、发布说明、文章的资料卡。
- `09-日记/`：每日观察、实验过程、学习记录、临时想法。
- `99-收件箱/`：暂存材料、待归档摘录、尚未分类的内容。

每个目录建议有一个 `说明.md` 或 `索引.md`，内容包括：

- 这个目录收什么。
- 不收什么。
- 常见笔记类型。
- 重要入口链接。

## 中文命名

优先使用用户未来会搜索的中文名称：

- `工具调用 Tool Calling.md`
- `OpenAI Responses API.md`
- `Claude Code.md`
- `Obsidian CLI.md`
- `RAG 评估.md`
- `知识库架构.md`
- `每日整理流程.md`

使用 `别名` 保存英文名、缩写、旧名称和常用叫法。

## 标签建议

说明性标签使用中文：

- `概念`
- `模型`
- `工具`
- `架构`
- `工作流`
- `评估`
- `来源`
- `提示词`
- `日记`
- `实验`
- `待整理`

命名空间标签可以保留英文或混合形式：

- `LLM/wiki`
- `LLM/RAG`
- `MCP`
- `API`

整理旧标签时，不要机械删除英文标签。若已有大量笔记使用 `architecture`，可先补充 `架构`，再由用户确认是否批量迁移。

## 概念笔记模板

```markdown
---
类型: 概念
状态: 草稿
标签:
  - LLM/wiki
  - 概念
别名: []
更新: YYYY-MM-DD
---

# 概念名称

## 用途

说明这篇笔记在知识库中的作用：它回答什么问题，适合从哪个主题进入，和哪些工作流相关。

## 摘要

两到四句话说明核心观点，尽量写成以后搜索时能直接读懂的答案。

## 说明

- 定义或机制。
- 使用场景。
- 限制或误区。

## 关联

- 上位主题：[[主题索引]]
- 相关概念：[[相关概念]]
- 来源：[官方文档](https://example.com)
```

## 实体笔记模板

```markdown
---
类型: 实体
状态: 草稿
标签:
  - LLM/wiki
  - 工具
别名: []
更新: YYYY-MM-DD
---

# 实体名称

## 用途

说明这篇笔记为什么存在：它用于记录某个模型、工具、提供方或 API 的能力、限制和接入方式。

## 摘要

它是什么，解决什么问题，什么时候应该考虑使用。

## 能力

- 主要能力。
- 集成方式。
- 已知限制。

## 操作说明

- 安装、命令、配置或兼容性说明。

## 关联

- 相关：[[相关笔记]]
- 来源：[官方文档](https://example.com)
```

## 日记笔记模板

```markdown
---
类型: 日记
状态: 记录
标签:
  - LLM/wiki
  - 日记
日期: YYYY-MM-DD
更新: YYYY-MM-DD
---

# YYYY-MM-DD 日记

## 用途

记录当天与 LLM Wiki、实验、阅读或工具实践相关的观察，之后可拆分为正式笔记。

## 今日记录

- 观察或实验。
- 问题或待验证想法。

## 待整理

- [ ] 可以拆成正式笔记的主题。
```

## 链接卫生

- 新笔记创建后立刻加入最近的索引。
- 两篇笔记互相重要时，尽量建立双向链接。
- 同义词、英文名、缩写放进 `别名`，不要创建重复页。
- 一个概念或实体只保留一个 canonical 页面。
- 重命名时检查旧链接、别名和相关索引。

## 适合检索的写法

- 顶部先写“用途”和“摘要”。
- 精确保留产品名、API 名、模型名、命令名。
- 示例短而可执行。
- 快速变化的信息要写日期和来源。
- 稳定概念和版本相关事实分开写。
