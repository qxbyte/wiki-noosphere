## 技术方案模板

```markdown
**背景**
[当前问题、业务目标、触发场景]

**目标**
- [目标 1]
- [目标 2]

**非目标**
- [本次明确不做的事情]

**现状**
- [相关模块、接口、数据流、限制]

**方案**
- [核心设计]
- [关键接口或数据结构]
- [兼容性处理]
- [错误处理和可观测性]

**备选方案**
- 方案 A：[优点 / 缺点 / 放弃原因]
- 方案 B：[优点 / 缺点 / 放弃原因]

**影响范围**
- 代码：[模块/文件]
- 数据：[表/字段/迁移]
- 接口：[API/事件/契约]
- 运维：[配置/CI/CD/发布]

**测试计划**
- [单元测试]
- [集成测试]
- [手动验证]
- [回归范围]

**风险与回滚**
- 风险：[说明]
- 缓解：[说明]
- 回滚：[说明]
```

## 任务拆解模板

```markdown
1. 建立上下文
   - 阅读 [模块/文件]
   - 确认 [接口/数据/测试]

2. 实现核心逻辑
   - 修改 [文件]
   - 保持 [兼容约束]

3. 接入调用路径
   - 修改 [入口/适配层]
   - 处理 [错误/权限/状态]

4. 补测试
   - 新增 [测试]
   - 覆盖 [边界/回归]

5. 验证与交付
   - 运行 [命令]
   - 输出 [PR 描述/发布说明]
```

## 子代理分工模板

```markdown
**主控工程师**
- 责任：[需求判断 / 方案 / 任务分派 / 接口契约 / 结果复核 / 整合 / 最终交付]
- 当前关键路径：[下一步必须由主控处理的事情]

**子代理 A：代码勘察**
- 目标：[要回答的问题]
- 只读范围：[目录/模块]
- 产出：[调用链 / 影响范围 / 同类实现 / 风险点]

**子代理 B：验证分析**
- 目标：[测试、CI 或复现问题]
- 命令：[可运行命令]
- 产出：[通过/失败结果、失败原因、建议修复点]

**子代理 C：实现**
- 目标：[具体 coding 任务]
- 写入范围：[明确文件/模块，避免与其他子代理冲突]
- 契约：[输入输出 / API / 状态 / 错误处理 / 兼容要求]
- 验证：[必须新增或运行的测试]
- 产出：[变更说明、测试结果、剩余风险]

**子代理 D：代码评审**
- 目标：[检查行为回归、边界、测试缺口]
- 范围：[文件/模块]
- 产出：[按严重程度排序的问题]

**整合标准**
- 主控复核所有子代理结论。
- 主控检查实现子代理是否遵守文件归属和接口契约。
- 冲突结论以代码事实和测试结果为准。
- 最终交付只包含已确认的变更、验证和风险。
```

## 测试计划模板

```markdown
**验证目标**
[要证明什么行为正确]

**自动化测试**
- [命令]：[覆盖内容]
- [命令]：[覆盖内容]

**手动验证**
- 步骤 1：[操作]
- 步骤 2：[操作]
- 期望结果：[结果]

**边界场景**
- [空数据/异常输入/权限不足/网络失败/并发/兼容性]

**未覆盖风险**
- [说明原因和后续建议]
```

## Review 回复模板

```markdown
已处理。

变更：
- [针对评论做了什么]
- [补充测试或验证]

验证：
- [命令/结果]
```

如果不同意 review 建议，使用：

```markdown
这里我建议暂时不这样改，原因是 [工程原因/兼容性/范围控制]。
我已补充 [替代方案/注释/测试] 来覆盖这个风险。
```
