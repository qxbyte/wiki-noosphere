## 这个 skill 是什么

`harness-hub` 是一个面向项目开发工程师的 Codex skill。它让 Codex 按照真实工程交付方式工作：理解需求、阅读代码、设计方案、实现修改、补充测试、运行验证、处理 CI/PR 反馈，并最终输出清楚的交付说明。

这里的 Harness Hub 指“把需求连接到可验证工程交付的统一入口”。它默认不是某个特定厂商工具的专属 skill；如果任务明确涉及 Harness.io、Harness CI/CD 或部署流水线，Codex 会把它作为 CI/CD 工程场景处理。

## 适合什么场景

适合使用这个 skill 的任务包括：

- 把产品需求或 issue 落地成代码实现。
- 在现有项目中新增功能、修复 bug、做小范围重构。
- 为开发任务补测试、补验证命令、补交付说明。
- 排查构建失败、测试失败、CI 失败、PR review 反馈。
- 设计技术方案、任务拆解、开发计划、测试计划。
- 准备 PR 描述、commit message、release note、交接文档。
- 作为项目开发工程师接手一个陌生仓库并推进任务闭环。

不适合单独使用的场景：

- 纯闲聊或只需要概念解释。
- 大范围组织流程咨询，但不涉及具体工程交付。
- 用户明确只要某个专门 skill，例如 PDF、图片生成、OpenAI 官方文档检索。

## 如何触发

可以直接在请求中写：

```text
使用 Harness Hub，帮我实现这个需求：...
```

也可以用中文工程交付类表达自然触发：

```text
作为项目开发工程师，帮我把这个 issue 落地。
```

```text
帮我修复这个 bug，并补测试和验证说明。
```

```text
帮我处理 PR review 里的问题，跑通测试后给我交付总结。
```

```text
帮我设计技术方案并拆成可执行开发任务。
```

如果希望显式使用 Codex 子代理，可以写：

```text
使用 Harness Hub，并结合 Codex 子代理并行处理这个需求：...
```

```text
使用 Harness Hub 多角色模式：主代理负责方案、分派、整合和交付，具体 coding 交给实现子代理。
```

## 推荐使用方式

最有效的请求通常包含四类信息：

1. **目标**：要实现什么、修复什么、优化什么。
2. **上下文**：相关文件、issue、PR、报错、日志、截图、用户路径。
3. **约束**：不能改什么、兼容要求、性能要求、交付时间、技术栈限制。
4. **验收**：如何判断完成、需要哪些测试、期望输出什么文档。

示例：

```text
使用 Harness Hub。目标：新增导出 CSV 功能。
上下文：现有导出逻辑在 src/export，列表页在 src/pages/orders。
约束：不要引入新依赖；保持现有权限模型。
验收：新增单元测试，能通过 npm test，并给我 PR 描述。
```

如果你只给一个简短需求，Codex 会先读仓库并做合理假设；遇到高风险不确定点时才会停下来问你。

## 典型工作流

### 1. 需求接收

Codex 会先识别任务类型：

- 新功能：确认入口、用户路径、数据契约和验收标准。
- Bug 修复：复现路径、失败原因、最小修复面、回归测试。
- 重构：行为保持、影响范围、迁移方式、测试保护。
- CI/PR 修复：失败日志、触发条件、相关 commit、验证命令。
- 技术方案：背景、目标、备选方案、风险、里程碑。

### 2. 代码勘察

Codex 会优先查看：

- 文件结构和构建配置。
- README、测试命令、开发脚本。
- 与需求相关的模块、接口、模型、测试。
- 同类功能和项目既有风格。
- 当前 git 状态，避免覆盖用户已有修改。

### 3. 方案设计

方案会尽量短而可执行，通常包含：

- 要改哪些模块。
- 为什么这样改。
- 测试与验证方式。
- 风险和假设。

简单任务会直接推进实现；复杂任务会先给你更明确的步骤。

### 4. 实现与验证

Codex 会按照现有项目习惯进行小范围修改，并根据风险选择验证层级：

- 小 bug：相关单测或目标测试文件。
- 共享逻辑：包级测试或更多回归测试。
- 构建配置：构建命令和 CI 等价命令。
- 前端交互：本地服务、浏览器检查、截图或交互验证。

### 5. 交付总结

完成后，Codex 会说明：

- 完成了什么。
- 修改了哪些关键文件。
- 跑了哪些测试或检查。
- 没跑成的验证和原因。
- 是否还有剩余风险或建议后续事项。

## 子代理协作模式

Harness Hub 可以和 Codex 子代理结合，适合任务比较大、信息面比较宽、验证成本比较高的工程工作。主代理相当于项目开发负责人或 tech lead，子代理相当于专项工程同伴。进入多角色模式后，具体 coding 默认交给实现子代理，主代理负责设计、分派、复核、集成和最终交付。

常见分工：

- **主控工程师**：理解需求、确定方案、定义模块边界和接口契约、分派实现任务、整合结果、给出最终交付。
- **代码勘察子代理**：搜索模块、梳理调用链、找同类实现、确认测试入口。
- **实现子代理**：在明确文件范围内完成具体 coding、测试补齐和局部验证。
- **验证子代理**：运行测试、分析失败日志、复现 bug、整理验证结论。
- **评审子代理**：检查边界条件、回归风险、缺失测试和行为不一致。
- **文档子代理**：生成 PR 描述、发布说明、交接文档和 review 回复草稿。

推荐在这些场景中启用：

- 陌生大型仓库，需要快速建立上下文。
- 一次改动横跨前端、后端、测试、CI 或文档。
- CI 失败日志很多，需要并行定位。
- PR review 评论较多，需要逐条处理。
- 高风险改动，需要独立评审视角。

不建议启用的场景：

- 单文件小改动。
- 需求还非常模糊，连目标都没有确定。
- 多个子代理必须同时编辑同一文件，容易产生冲突。
- 当前下一步完全依赖一个核心判断，需要主代理先完成方案判断。

示例：

```text
使用 Harness Hub，并结合 Codex 子代理处理。
主代理负责方案、任务分派、结果复核和最终整合；
子代理 A 勘察现有实现；
子代理 B 在指定文件范围内实现修复并补测试；
子代理 C 分析测试和 CI；
子代理 D 做代码审查。
目标：修复订单导出在空数据时崩溃的问题。
验收：补回归测试并给出 PR 描述。
```

## 常用提示词

### 新功能开发

```text
使用 Harness Hub，帮我在当前项目实现 [功能]。
要求：[约束]。
验收：[测试/交互/文档要求]。
```

### Bug 修复

```text
使用 Harness Hub，修复这个问题：
[现象/报错/复现步骤]
请定位原因、实现修复、补回归测试，并总结验证结果。
```

### CI 失败处理

```text
使用 Harness Hub，帮我处理 CI 失败：
[日志或失败命令]
请判断根因，做最小修复，并运行对应验证。
```

### PR Review 处理

```text
使用 Harness Hub，处理这些 review comments：
[评论列表]
请逐条解决，必要时补测试，最后给我回复建议。
```

### 技术方案

```text
使用 Harness Hub，给 [需求] 设计技术方案。
请包含方案选择、影响范围、任务拆解、风险、测试计划和交付标准。
```

### 交付文档

```text
使用 Harness Hub，基于这次改动生成 PR 描述和发布说明。
```

### 子代理并行开发

```text
使用 Harness Hub，并结合 Codex 子代理并行处理：
目标：[需求]
分工：主代理负责方案、分派、复核和整合；实现子代理负责具体 coding；其他子代理负责代码勘察、测试验证、代码审查。
验收：[测试/交付要求]
```

## 协作约定

- Codex 会优先直接动手完成任务，而不是只给建议。
- 只有当你明确要求使用子代理、subagent、并行代理或多角色模式时，Harness Hub 才会启用子代理协作。
- 对高风险操作，例如删除文件、重置分支、覆盖用户改动，会先请求确认。
- 如果仓库有未提交改动，Codex 会默认认为是你的工作，除非你明确要求清理。
- 如果命令因为权限、网络或环境限制失败，Codex 会说明失败原因，并在需要时请求授权。
- 如果你要求“review”，Codex 会进入代码审查视角，优先输出问题和风险。

## 输出物说明

根据任务需要，Codex 可以产出：

- 代码修改。
- 单元测试、集成测试、回归测试。
- 技术方案。
- 开发任务拆解。
- 测试计划。
- PR 描述。
- Review 回复建议。
- Release note。
- 交接文档。

详细模板见 `references/templates.md`。

## 最佳实践

- 把失败日志、截图、issue 链接或 PR 评论直接贴给 Codex。
- 明确哪些文件或模块不能改。
- 明确是否需要提交 commit、推送分支或创建 PR。
- 对大型需求，先让 Codex 输出任务拆解，再逐项实现。
- 对线上风险任务，要求 Codex 列出回滚策略和发布验证点。
